Why this is novel
Most student projects distribute one stage. This system distributes all three: the crawl (MPI workers), the index build (MapReduce), and the query execution (federated shards). The entire pipeline is parallel end-to-end — the same claim Apache Hadoop and Elasticsearch make about their architectures.

PDC concepts implemented
Formal distributed computing theory made concrete and measurable
Task Parallelism
MPI Master-Worker Crawling
Six worker processes crawl web pages simultaneously using MPI message passing. The master maintains the URL frontier and dispatches tasks via tagged messages. This is Flynn's MIMD (Multiple Instruction, Multiple Data) taxonomy — each worker executes different code on different URLs in parallel.
Flynn (1966) — Very High Speed Computing Systems
Data Partitioning
Consistent Hash Ring
URLs are routed to workers by domain using a consistent hash ring with 150 virtual nodes per worker. When a worker dies, only 1/N of assignments change — not the entire mapping. This is the core innovation of Amazon Dynamo: hash ring partitioning minimises rebalancing cost during node churn.
Karger et al. (1997) — Consistent Hashing and Random Trees
Fault Tolerance
Heartbeat + Task Requeue
Workers send a heartbeat every 4 seconds. If no heartbeat arrives within 15 seconds, the master marks that worker dead and re-queues its in-progress URL. This is the standard failure detector model from distributed systems theory — trading accuracy (false positives) for completeness (never misses a real failure).
Chandra & Toueg (1996) — Unreliable Failure Detectors
Crash Recovery
Checkpointing
The master saves the URL frontier and visited set to disk every 200 pages. On restart, it resumes from the last checkpoint. This is the Write-Ahead Log (WAL) pattern applied to a distributed job — the same mechanism databases use for transaction durability and Hadoop uses for job recovery.
Gray & Reuter (1992) — Transaction Processing
Probabilistic Structures
Bloom Filter Deduplication
A bit array of size m with k hash functions answers "have I seen this URL?" in O(k) time using 56× less memory than a Python set. The false positive rate follows: p ≈ (1 − e^(−kn/m))^k. Measured empirically: 0.94% FP rate vs 1.0% target — matching theory within 6%.
Bloom (1970) — Space/Time Trade-offs in Hash Coding
Epidemic Algorithms
Gossip Protocol
Workers periodically exchange Bloom filter bytes via bitwise OR. Information spreads like a virus — each round, each node's knowledge grows. Convergence is guaranteed in O(log N) rounds. This is how Amazon DynamoDB propagates cluster membership and how Bitcoin propagates transactions without a central server.
Demers et al. (1987) — Epidemic Algorithms for Replicated Database Maintenance
Synchronisation
Token Bucket Rate Limiting
The master holds one token bucket per domain. Workers must request permission before fetching — coordinated shared state across 6 processes via message passing. This demonstrates the classic distributed synchronisation problem: how do multiple independent processes share a resource without locks?
Turner (1986) — New Directions in Communications
Data Parallelism
MapReduce Indexing
Six mapper processes tokenise 22,740 documents in parallel, emitting 85 million (word, doc_id) pairs. A single reducer aggregates them into 830,928 unique terms. This directly implements Dean & Ghemawat's MapReduce model: the same programming abstraction as Apache Hadoop, built from scratch with MPI.
Dean & Ghemawat (2004) — MapReduce: Simplified Data Processing on Large Clusters
Iterative Computation
PageRank Power Iteration
PageRank is computed via 20 iterations of power iteration on the crawled link graph. Each iteration updates all node scores in parallel. The algorithm converges when Σ|new − old| < 1e-6. This is the same iterative pattern used by Apache Spark's GraphX for distributed graph processing.
Brin & Page (1998) — The Anatomy of a Large-Scale Hypertextual Web Search Engine
Distributed Querying
Federated Shard Search
The inverted index is split into N shards by doc_id range. Each shard is searched independently via MPI. A coordinator merges N sorted result lists using a max-heap in O(K·N·log N) time. This closes the loop — crawling, indexing, AND querying are all distributed. The same architecture as Apache Solr Cloud and Elasticsearch.
Callan et al. (1995) — Searching Distributed Collections with Inference Networks
Semantic Search
BERT + FAISS Embeddings
Documents are encoded as 384-dimensional vectors using the all-MiniLM-L6-v2 sentence transformer. Queries are answered via approximate nearest neighbour search (FAISS IndexFlatIP) using inner product similarity (equivalent to cosine similarity after L2 normalisation). This is how every modern production search engine handles semantic understanding.
Johnson et al. (2019) — Billion-scale Similarity Search with GPUs (FAISS)
Eventual Consistency
Collaborative Search Model
A shared in-memory relevance model is updated by all users simultaneously. Scores decay with a half-life of 5 minutes using exponential decay: score × e^(−t·ln2/τ). Updates propagate to all connected clients via SSE. This is an AP system in CAP theorem terms — available and partition-tolerant, eventually consistent.
Brewer (2000) — Towards Robust Distributed Systems (CAP Theorem)

Performance results
Measured empirically, not estimated
Speedup formula (Amdahl's Law):
S(p) = T₁ / Tₚ # T₁ = 128.5s (sequential baseline) # Tₚ = parallel time with p workers S(2) = 128.5 / 73.5 = 1.75× S(3) = 128.5 / 41.8 = 3.07× ← super-linear S(6) = ~770 / 128.5 = ~6.00× E(p) = S(p) / p × 100% E(3) = 3.07 / 3 × 100 = 102.5%

Efficiency above 100% is super-linear speedup — a known phenomenon in I/O-bound distributed systems where parallel network requests overlap in ways that sequential execution cannot. Each worker maintains independent TCP connections to Wikipedia, so 3 workers get 3× the I/O throughput simultaneously.

Bloom filter mathematics:
# Optimal parameters m = -n·ln(p) / (ln 2)² k = (m/n)·ln 2 # At n=500,000, p=0.001: m = 7,188,793 bits (0.86 MB) k = 9 hash functions # Actual FP rate at 3.75M URLs: p_actual ≈ 3.58% (exceeded capacity) # Fix: capacity=10,000,000 m_new = 14.4M bits (1.71 MB)

The overnight run inserted 3.75M URLs into a filter sized for 500K — causing FP rate to exceed target. The fix is to pre-size the filter for expected frontier size, not expected crawl size. Frontier can be 10–100× larger than pages actually crawled.

MPI message protocol
Tagged message passing — the nervous system of the distributed crawler

All inter-process communication uses tagged MPI messages. The master uses comm.iprobe() (non-blocking) in a polling loop so it can interleave heartbeat checking, token granting, gossip exchange, and task assignment without blocking on any single operation. Workers use comm.probe() to check the tag before consuming a message.

TAG_RESULT = 3Worker → Master(doc_id, url, title, text, snippet, links, depth)Crawled page data
TAG_DONE = 4Master → WorkerNoneNo more work — shut down cleanly

Search ranking explained
How a query becomes a ranked list of results
Step 1
BM25 Relevance (75%)
Each query term scores documents using BM25 with k₁=1.5, b=0.75. IDF penalises common terms; TF saturation prevents one term from dominating. PageRank blended at 20% weight.
Robertson & Jones (1976)
Step 2
BERT Semantics (25%)
Query is encoded to a 384D vector. FAISS retrieves the 40 nearest document vectors by inner product. Semantic score is normalised to [0,1] and blended with BM25 at 25% weight.
Reimers & Gurevych (2019)
Step 3
Click Boost (adaptive)
Documents clicked historically for similar queries receive a log-scale boost: 1.0 + 0.1·log₂(clicks + 1). The system learns from user behaviour over time without retraining any model.
Joachims et al. (2007)
Enhancement
Spell Correction
Mistyped query terms are corrected using Levenshtein edit distance ≤ 2 against the 830K-term trie. Candidates are filtered by first character to reduce the search space from O(V) to O(V/26).
Levenshtein (1966)
Enhancement
Query Expansion
Query terms are expanded with co-occurring terms from the index and curated synonyms. The expanded query is passed to BM25 while the original is used for BERT — preventing semantic drift from expansion noise.
Rocchio (1971) — Relevance Feedback
Real-time
Crowd Boost (collaborative)
In collaborative mode, documents clicked by other users for similar queries receive a real-time DCG-weighted boost: position_weight = 1/log₂(pos + 2). Scores decay with half-life τ=300s.
Järvelin & Kekäläinen (2002) — DCG

CAP theorem analysis
Every distributed design decision is a trade-off
CP System
The Crawler (Consistent + Partition-tolerant)
The URL visited set is centralised in the master — strongly consistent. No URL will ever be crawled twice (barring Bloom filter false positives). During a network partition, the system stops crawling (sacrifices Availability) rather than risk duplicates. Dead workers' tasks are requeued, not continued independently.
AP System
The Collaborative Model (Available + Partition-tolerant)
The shared relevance model accepts updates from all users without coordination. If two users click the same document simultaneously, both boosts are applied (last-write-wins for the score, not the data). The system remains available during any partition — eventual consistency via score decay brings it back to a consistent state.
Trade-off
Gossip Protocol (Eventual Consistency)
Workers exchange Bloom filters periodically rather than synchronously. During the window between gossip rounds, two workers may both attempt to crawl the same URL. This is a deliberate choice: the performance benefit of asynchronous gossip outweighs the cost of occasional duplicate crawls at our scale.
Storage
SQLite WAL (MVCC Concurrency)
SQLite in Write-Ahead Log mode implements Multi-Version Concurrency Control: readers never block writers, writers never block readers. Multiple Flask threads read the index simultaneously while the incremental indexer writes new documents. This is the same concurrency model as PostgreSQL and Oracle.
